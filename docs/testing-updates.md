# 测试浏览器插件更新与数据迁移

在重构项目或开发涉及本地数据格式变动的版本时，我们通常需要编写迁移脚本（如 `migration.ts`），并在 `chrome.runtime.onInstalled (reason: 'update')` 生命周期中触发。

如何在发布到应用商店前，在本地完美测试“无感更新”和数据迁移逻辑？这里提供两种方法。

## 方法一：使用内置的“导入旧版文件”工具（推荐，最快捷）

如果你只想验证数据转换逻辑是否正确，而不想折腾浏览器插件的安装与 ID 问题，可以直接使用页面上提供的导入测试功能。

1. **导出旧版数据**：在旧版插件的设置页中，导出旧的配置文件（通常是 JSON 格式）。
2. **清理新版数据**：在当前新版插件的“导入/导出（Config Manager）”页面中，点击 `Clear Configuration`，清空新版当前的所有本地存储，防止数据互相干扰。
3. **导入并迁移**：
   - 点击 `Import Legacy File`，选择你刚才导出的旧版 JSON 文件。此时系统会强制将这批旧版原始数据写入到浏览器的本地存储中。
   - 点击 `Run Migration Test` 按钮，手动触发一次迁移脚本（`runFullMigration`）。
4. **校验结果**：查看下方的 `Import Logs` 日志是否报错，并前往对应设置页（如模型设置），检查所有历史配置（例如遗留的模型图标、不一致的 URL 等）是否都已完美转化为新版支持的格式。

---

## 方法二：模拟真实的浏览器插件静默更新（Hardcore 黑科技）

如果你想端到端测试“当用户浏览器在后台自动下载新代码并覆盖旧代码的全过程”，就需要模拟真实的 `update` 生命周期。

**难点**：在 Chrome 的开发者模式下，加载未打包插件（Unpacked Extension）时，插件的 ID 是根据**文件夹的绝对路径**生成的。由于新旧版代码（比如 Vue 版和 React 版）在不同的文件夹，直接分别加载会被识别为两个独立的插件，导致它们的 `browser.storage.local` 是物理隔离的，无法触发覆盖更新。

**解决方案（鸠占鹊巢法）**：

1. **定位旧巢**：找到你当前在 `chrome://extensions/` 页面里正在运行的、含有真实旧版数据的**旧版插件文件夹**路径。
2. **腾空旧巢**：打开该文件夹，把里面的文件**全部删掉**。
   > **注意**：Chrome 的本地数据库储存在浏览器全局的 `userData` 目录下，与插件文件夹里的代码文件无关，只要文件路径还在，数据就不会丢。
3. **注入新代码**：回到你的新版项目，执行 `npm run build`。将打包生成目录（如 `.output/chrome-mv3`）下的所有文件，**全部复制粘贴**到刚才被腾空的旧版插件文件夹中。
4. **触发更新**：回到浏览器的 `chrome://extensions/` 页面，找到原本的旧插件卡片（由于路径没变，ID 仍是旧的），点击卡片上的**刷新（Reload）按钮**。

**测试结果**：
Chrome 发现路径未变，认定其为同一个插件（ID 保持一致）；但在重新读取文件后，发现版本号或代码内容发生改变，于是会立刻触发 `chrome.runtime.onInstalled` 并携带 `reason: 'update'` 参数。
此时，后台脚本开始静默运行 `migration.ts`，新代码完美继承并清洗旧插件的本地数据库。当你再次打开新版插件设置页时，所有数据均已无缝迁移完成！
